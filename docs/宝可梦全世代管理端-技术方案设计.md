## 1. 项目背景与目标

### 1.1 现状痛点

- PKHeX 作为最权威的宝可梦存档编辑器，仅支持客户端模式运行。
- 导出的宝可梦为单个二进制文件（`.pk1` ~ `.pk7`），缺乏即时可视化预览能力。
- 无法实现跨设备、随时随地整理自己的宝可梦收藏。
- 没有在线「宝可梦银行」（Pokémon Bank）概念，用户散落在各存档中的宝可梦无法集中管理。
- 编辑-导出-验证流程繁琐，修改后需要反复导入游戏确认。

### 1.2 项目目标

打造一个 **B/S 架构** 的宝可梦在线管理平台，实现：

1. **用户体系** — 每个用户拥有独立账号和个人宝可梦银行（Pokémon Bank）。
2. **存档在线解析与可视化** — 上传游戏存档（`.sav`/`.dsv`），网页端以「箱子网格」形式直观展示存档内全部宝可梦。
3. **拖拽式管理** — 通过鼠标拖拽，在存档箱子与个人银行之间、箱子与箱子之间自由移动宝可梦。
4. **在线编辑 + 合法性仲裁** — 提供与 PKHeX 同级别的全属性编辑面板，所有修改经 PKHeX.Core 合法性验证。
5. **存档持久化与再使用** — 修改后的存档保存于服务端，可随时重新加载继续编辑，也可导出为合法存档文件供游戏使用。
6. **跨世代兼容** — 天然支持 GBA (Gen3)、NDS (Gen4/5)、3DS (Gen6/7) 全部主系列版本。

**架构原则**:
- 前端只负责渲染与交互，所有二进制解析、结构读写、合法性检查均由后端 PKHeX.Core 完成。
- 宝可梦数据在前后端之间以规范化 JSON 传输。
- PostgreSQL 以关系表 + JSONB 混合模式存储，兼顾结构化查询与世代差异字段的灵活性。

### 4.3 拖拽交互设计

使用 `@dnd-kit` 实现三种拖拽操作：

| 拖拽类型 | 源 | 目标 | 行为 |
|----------|-----|------|------|
| **存档→银行** | 箱子中的宝可梦格 | 银行区域 | 将该宝可梦复制/移动至个人银行 |
| **银行→存档** | 银行中的宝可梦 | 箱子中的空格 | 将银行宝可梦放入存档指定位置 |
| **箱子→箱子** | 箱子A中的宝可梦 | 箱子B中的空格 | 在存档内部移动宝可梦位置 |
| **存档内交换** | 箱子A中的宝可梦 | 箱子A中已有宝可梦的格 | 交换两只宝可梦位置 |

**拖拽视觉反馈**:
- 拖拽时显示半透明宝可梦精灵 + 名称的覆盖层。
- 可放置的目标格子高亮（绿色边框）。
- 不可放置的目标（如已满箱子且有宝可梦的格）显示红色边框。
- 支持多选批量拖拽（Shift/Ctrl 多选后一起拖入银行）。

### 5.2 核心服务实现逻辑

#### 5.2.1 存档解析服务 (ParseService)

```csharp
public class ParseService
{
    /// <summary>
    /// 解析存档文件，返回所有箱子及内部宝可梦的结构化数据
    /// </summary>
    public SaveFileDetailDto ParseSaveFile(byte[] saveData, string fileName)
    {
        // PKHeX.Core 自动识别存档格式并实例化对应的 SAV 对象
        var sav = SaveUtil.GetVariantSAV(saveData);
        if (sav == null)
            throw new BusinessException("不支持的存档格式或文件已损坏");

        var boxes = new List<BoxDto>();
        int boxIndex = 0;

        foreach (var box in sav.BoxData)
        {
            var slots = new List<BoxSlotDto>();
            for (int slot = 0; slot < box.Length; slot++)
            {
                var pkm = box[slot];
                if (pkm.Species == 0) // 空格子
                {
                    slots.Add(new BoxSlotDto { SlotIndex = slot, IsEmpty = true });
                }
                else
                {
                    slots.Add(new BoxSlotDto
                    {
                        SlotIndex = slot,
                        IsEmpty = false,
                        Pokemon = MapToPokemonDto(pkm)
                    });
                }
            }
            boxes.Add(new BoxDto
            {
                BoxIndex = boxIndex++,
                BoxName = sav.GetBoxName(boxIndex - 1),
                Slots = slots,
                Capacity = box.Length
            });
        }

        return new SaveFileDetailDto
        {
            Generation = sav.Generation,
            GameVersion = sav.Version,
            TrainerName = sav.OT,
            TrainerID = sav.TID,
            SecretID = sav.SID,
            Boxes = boxes,
            PlayTime = sav.PlayedTime,
            // ... 其他存档元数据
        };
    }

    /// <summary>
    /// PKM 对象 → 前端可消费的 DTO
    /// </summary>
    private PokemonDto MapToPokemonDto(PKM pkm)
    {
        return new PokemonDto
        {
            Species = pkm.Species,
            SpeciesName = GameInfo.Strings.Species[pkm.Species],
            Nickname = pkm.Nickname,
            Gender = (int)pkm.Gender,
            Level = pkm.CurrentLevel,
            Nature = (int)pkm.Nature,
            NatureName = GameInfo.Strings.Natures[pkm.Nature],
            Ability = pkm.Ability,
            AbilityName = GameInfo.Strings.Ability[pkm.Ability],
            IsShiny = pkm.IsShiny,
            IsEgg = pkm.IsEgg,
            HeldItem = pkm.HeldItem,
            IVs = new[] { pkm.IV_HP, pkm.IV_ATK, pkm.IV_DEF, pkm.IV_SPA, pkm.IV_SPD, pkm.IV_SPE },
            EVs = new[] { pkm.EV_HP, pkm.EV_ATK, pkm.EV_DEF, pkm.EV_SPA, pkm.EV_SPD, pkm.EV_SPE },
            Moves = new[]
            {
                new MoveDto { MoveId = pkm.Move1, MoveName = GameInfo.Strings.Move[pkm.Move1] },
                new MoveDto { MoveId = pkm.Move2, MoveName = GameInfo.Strings.Move[pkm.Move2] },
                new MoveDto { MoveId = pkm.Move3, MoveName = GameInfo.Strings.Move[pkm.Move3] },
                new MoveDto { MoveId = pkm.Move4, MoveName = GameInfo.Strings.Move[pkm.Move4] }
            },
            PID = pkm.PID,
            TID = pkm.TID,
            SID = pkm.SID,
            Ball = pkm.Ball,
            MetLocation = pkm.Met_Location,
            MetLevel = pkm.Met_Level,
            MetDate = pkm.MetDate,
            IsValid = true, // 上传原始数据默认为合法
            // 将完整二进制数据序列化为 Base64 用于重建
            PkmDataBase64 = Convert.ToBase64String(pkm.DecryptedPartyData)
        };
    }
}
```

#### 5.2.2 编辑与合法性校验服务 (PokemonEditService)

```csharp
public class PokemonEditService
{
    /// <summary>
    /// 将前端编辑后的 JSON 数据应用到 PKM 对象并执行合法性校验
    /// </summary>
    public EditResultDto ApplyEdits(PKM original, PokemonEditRequest request)
    {
        // 逐字段应用修改
        if (request.Species.HasValue)
            original.Species = request.Species.Value;

        if (request.Nickname != null)
            original.Nickname = request.Nickname;

        if (request.Nature.HasValue)
            original.SetNature(request.Nature.Value);

        if (request.IVs != null)
            original.SetIVs(request.IVs);

        if (request.EVs != null)
            original.SetEVs(request.EVs);

        if (request.Moves != null)
        {
            for (int i = 0; i < Math.Min(4, request.Moves.Length); i++)
                original.SetMove(i, request.Moves[i]);
        }

        if (request.IsShiny.HasValue)
            original.SetShiny(request.IsShiny.Value);

        if (request.Ability.HasValue)
            original.Ability = request.Ability.Value;

        if (request.Ball.HasValue)
            original.Ball = request.Ball.Value;

        // ... 其他字段

        // PKHeX.Core 核心合法性分析
        var legality = new LegalityAnalysis(original);

        return new EditResultDto
        {
            IsValid = legality.Valid,
            Report = legality.Valid ? null : legality.Report(),
            Judgements = legality.Results.Select(r => new JudgementDto
            {
                Identifier = r.Identifier,
                Judgement = r.Judgement.ToString(),
                Comment = r.Comment
            }).ToList(),
            UpdatedPokemon = MapToPokemonDto(original) // 返回最新状态
        };
    }
}
```

#### 5.2.3 存档重建服务 (SaveFileService)

```csharp
public class SaveFileService
{
    /// <summary>
    /// 将用户修改后的箱子数据写回存档文件，生成可下载的合法存档
    /// </summary>
    public byte[] RebuildSaveFile(Guid saveFileId, List<BoxUpdateDto> updatedBoxes)
    {
        // 从数据库加载原始存档二进制
        var saveFile = _repository.GetSaveFile(saveFileId);
        var sav = SaveUtil.GetVariantSAV(saveFile.RawData);

        foreach (var boxUpdate in updatedBoxes)
        {
            var targetBox = sav.BoxData[boxUpdate.BoxIndex];

            foreach (var slotUpdate in boxUpdate.Slots)
            {
                if (slotUpdate.PokemonId.HasValue)
                {
                    // 从银行或存档其他位置获取宝可梦数据
                    var pokemonData = _bankRepository.GetPokemonData(slotUpdate.PokemonId.Value);
                    var pkm = EntityConverter.ConvertToPkm(pokemonData, sav.Generation);
                    targetBox[slotUpdate.SlotIndex] = pkm;
                }
                else if (slotUpdate.IsEmpty)
                {
                    // 清空格子（生成对应世代的空白宝可梦）
                    targetBox[slotUpdate.SlotIndex] = sav.BlankPKM;
                }
            }
        }

        // PKHeX.Core 重新计算存档校验和并写入
        // SaveUtil 或 SAV.Write() 内部自动处理
        return sav.Write();
    }

    /// <summary>
    /// 保存存档到服务端（用于后续继续编辑）
    /// </summary>
    public void PersistSave(Guid saveFileId, byte[] updatedSaveData)
    {
        _repository.UpdateSaveFileData(saveFileId, updatedSaveData);
    }
}
```

## 6. PostgreSQL 数据库设计

### 6.1 设计原则

- **关系表 + JSONB 混合**: 需要频繁查询、筛选、排序的字段（物种、世代、闪光等）存为独立列；完整宝可梦数据以 JSONB 存储，适应世代间字段差异。
- **外键约束 + 级联删除**: 用户删除时级联清理其所有存档和银行数据。
- **GIN 索引**: 对 JSONB 列建 GIN 索引加速 JSON 内部查询。
- **BYTEA 存储存档**: 原始存档文件以 BYTEA 存储在 PostgreSQL 中，同时可选文件系统备份。

## 7. 核心功能模块与实现方案

### 7.1 用户注册与登录

**注册流程**:
1. 前端表单校验：用户名 3-50 字符、邮箱格式、密码 8+ 字符含大小写+数字。
2. 后端 `POST /api/auth/register` 接收，检查用户名/邮箱唯一性。
3. BCrypt 哈希密码（cost factor = 12）。
4. 插入 `users` 表，返回 JWT access_token + refresh_token。

**登录流程**:
1. `POST /api/auth/login` 提交 `{username, password}`。
2. 查询 `users` 表，BCrypt 验证密码。
3. 签发 JWT: payload 包含 `sub` (user_id), `username`, `iat`, `exp`。
4. access_token 有效期 2 小时，refresh_token 有效期 7 天。

**JWT 中间件**:
```csharp
// 每个请求从 Authorization Header 提取 Bearer Token
// 验证签名和有效期，将 user_id 注入 HttpContext.Items["UserId"]
// Controller 通过 [FromServices] UserContext 获取当前用户
```

### 7.3 拖拽操作：存档 ↔ 银行

这是项目的核心交互。拖拽分为三种原子操作：

#### 7.3.1 从存档箱子拖入银行 (Save → Bank)

```
1. 用户在存档箱子中拖拽一只宝可梦到「我的银行」区域
2. 前端触发 POST /api/bank/from-save
   Body: { saveFileId, boxIndex, slotIndex }

3. 后端处理:
   a. 查询 save_box_pokemon 获取 pokemon_json
   b. INSERT INTO bank_pokemon (
        user_id, species, ..., pokemon_json, pkm_data_base64,
        source: 'save_import', source_save_id
      )
   c. UPDATE save_box_pokemon SET is_empty = TRUE,
        pokemon_json = NULL, species = NULL
      WHERE save_file_id = ? AND box_index = ? AND slot_index = ?
   d. UPDATE save_files SET is_modified = TRUE

4. 返回新增的 bank_pokemon.id 和更新后的存档状态
5. 前端更新本地状态：箱子格子变空，银行列表增加新条目
```

**行为选项**: 可在前端提供「移动」vs「复制」切换按钮，默认行为为移动。

#### 7.3.2 从银行拖入存档箱子 (Bank → Save)

```
1. 用户从银行拖拽宝可梦到存档箱子的空格或已有宝可梦的格
2. 前端触发 POST /api/save-file/{id}/move-from-bank
   Body: { bankPokemonId, targetBoxIndex, targetSlotIndex }

3. 后端处理:
   a. 查询 bank_pokemon 获取数据
   b. 检查目标位置状态:
      - 如果 targetSlot 为空 → 直接放置
      - 如果 targetSlot 有宝可梦 → 先将被占位的宝可梦写入一个空格，再放置
                                   (或交换: 占位宝可梦移入银行)
   c. UPDATE save_box_pokemon SET
        is_empty = FALSE, pokemon_json = ..., species = ...,
        source_bank_id = bankPokemonId
      WHERE save_file_id = ? AND box_index = ? AND slot_index = ?
   d. 如果是移动模式: DELETE FROM bank_pokemon WHERE id = ?
      如果是复制模式: 保留银行记录
   e. UPDATE save_files SET is_modified = TRUE

4. 返回更新后的存档状态
```

#### 7.3.3 存档内箱子间移动 (Box → Box)

```
1. 用户在存档内从一个箱子拖到另一个箱子
2. 前端触发 POST /api/save-file/{id}/move-slot
   Body: {
     fromBoxIndex, fromSlotIndex,
     toBoxIndex, toSlotIndex
   }

3. 后端处理:
   a. 读取 fromSlot 数据
   b. 如果 toSlot 有宝可梦:
      - 交换: fromSlot 数据写入 toSlot, toSlot 原数据写入 fromSlot
      或
      - 移动: toSlot 数据移入同箱第一个空格 (如果 toSlot 为空则直接放置)
   c. 执行两个 UPDATE 语句 (事务)
   d. UPDATE save_files SET is_modified = TRUE

4. 返回更新后的存档状态
```

### 7.4 保存存档与导出

#### 7.4.1 服务端持久化保存

```
POST /api/save-file/{id}/save
→ 后端从 save_box_pokemon 表读取该存档所有箱子的当前数据
→ 通过 SaveFileService 重建存档二进制
→ 更新 save_files.raw_save_data (新二进制覆盖旧值)
→ 更新 save_files.updated_at, is_modified = FALSE
→ 返回成功状态
```

用户关闭浏览器后，下次登录仍可打开存档继续编辑。

#### 7.4.2 导出下载

```
GET /api/save-file/{id}/download
→ 后端重建存档二进制（如果 is_modified = TRUE）
→ 返回 File(bytes, "application/octet-stream", "filename.sav")
→ 下载的文件可直接导入对应游戏或 PKHeX 继续使用
```

### 7.5 宝可梦在线编辑

编辑流程整合前端编辑面板与后端校验：

```
1. 用户点击存档中的宝可梦 → 右侧展开编辑面板
2. 编辑面板加载该宝可梦的完整 pokemon_json 数据
3. 用户在表单中修改属性（物种、招式、性格等）
4. 点击「保存修改」
   →
5. POST /api/pokemon/{id}/edit
   Body: { species, nickname, nature, ivs: [...], evs: [...], moves: [...], ... }
   →
6. 后端 PokemonEditService:
   a. 从 pokemon_json.pkm_data_base64 重建 PKM 对象
   b. 逐一应用修改
   c. new LegalityAnalysis(pkm) → 合法性报告
   d. 如果合法 → 更新 save_box_pokemon.pokemon_json
   e. 如果不合法 → 返回详细违规报告（具体到哪个字段不合法）
   →
7. 前端展示结果:
   - 合法: 绿色对勾 + 更新预览
   - 非法: 红色警告列表 + 拒绝修改
```

### 9.3 拖拽结束逻辑 (handleDragEnd)

```typescript
type DragSource =
  | { type: 'save-box'; boxIndex: number; slotIndex: number; pokemonId: string }
  | { type: 'bank'; bankPokemonId: string };

type DragTarget =
  | { type: 'save-box'; boxIndex: number; slotIndex: number }
  | { type: 'bank' };

async function handleDragEnd(event: DragEndEvent) {
  const { active, over } = event;
  if (!over) return; // 没有有效目标，取消

  const source = parseSource(active);   // 从 active.id 解析来源
  const target = parseTarget(over);     // 从 over.id 解析目标

  // Case 1: 存档箱子 → 银行
  if (source.type === 'save-box' && target.type === 'bank') {
    await bankService.moveFromSave({
      saveFileId: currentSaveId,
      boxIndex: source.boxIndex,
      slotIndex: source.slotIndex,
    });
    // 刷新存档和银行状态
    refreshBoth();
  }

  // Case 2: 银行 → 存档箱子
  else if (source.type === 'bank' && target.type === 'save-box') {
    await saveFileService.moveFromBank(currentSaveId, {
      bankPokemonId: source.bankPokemonId,
      targetBoxIndex: target.boxIndex,
      targetSlotIndex: target.slotIndex,
    });
    refreshBoth();
  }

  // Case 3: 存档内部移动/交换
  else if (source.type === 'save-box' && target.type === 'save-box') {
    await saveFileService.moveSlot(currentSaveId, {
      fromBoxIndex: source.boxIndex,
      fromSlotIndex: source.slotIndex,
      toBoxIndex: target.boxIndex,
      toSlotIndex: target.slotIndex,
    });
    refreshSave();
  }
}
```

## 11. 方案优势与可扩展性

### 11.1 核心优势

| 维度 | 说明 |
|------|------|
| **真实银行体验** | 首创网页端「宝可梦银行」+「存档箱子」拖拽交互，复原 3DS Pokémon Bank 核心体验 |
| **合法性零妥协** | 所有数据操作均通过 PKHeX.Core 合法性仲裁，与 PKHeX 桌面端完全一致 |
| **存档可复用** | 修改后的存档保存于服务端，随时重新编辑、随时下载，告别一次性流程 |
| **跨世代兼容** | 天然支持 GBA→3DS 全部主系列版本，PKHeX.Core 自动处理世代差异 |
| **跨设备访问** | 浏览器即可使用，PC/平板/手机均可操作 |
| **数据持久化** | PostgreSQL 事务保证数据一致性，存档和银行数据永不丢失 |
| **可扩展架构** | 分层清晰，后续可轻松添加组队分析、对战模拟、世代迁移等高级功能 |

### 11.2 后续迭代方向

- **Switch 世代支持** (Gen8/Gen9): 当 PKHeX.Core 支持 SwSh/BDSP/PLA/SV 后，只需更新 NuGet 包即可扩展
- **多人协作**: 对战队伍分享、银行宝可梦公开交易、社区图鉴
- **图鉴系统**: 全国图鉴完成度追踪、形态收集进度
- **对战分析**: 能力值计算器、伤害计算、组队合理性分析
- **世代迁移**: 自动处理跨世代宝可梦数据格式转换（Gen3→Gen4→...→Gen7）
- **移动端 App**: React Native 复用现有 API，原生拖拽体验
- **管理后台**: 管理员用户管理、数据统计、异常宝可梦审计
